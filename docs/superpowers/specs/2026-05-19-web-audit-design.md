## 1. Goal

Land the founder console's Audit feature: a single page with three sub-tabs (Activity, Escalations, Traces) that surfaces the per-org `audit_log` table without exposing the schema. The page accepts deep links from Tasks / Agents / Threads with a pre-filtered view, and adds three reusable patterns to the design system: `SubTabBar`, an expandable-row pattern, and a `TraceTree` renderer with cost annotations.

## 2. Non-goals

- **Writing audit entries from the UI.** All entries are system-generated by `src/infrastructure/audit_logger.py`.
- **Real-time SSE.** Per umbrella §5.5, Audit polls (`refetchInterval: 60_000`). The SSE budget is reserved for Threads + Tasks.
- **Charts.** No sparklines, no totals dashboard. That belongs in PR 12.
- **Bulk operations.** No multi-select, no export.
- **Custom date range picker.** Three quick chips (today / this week / all) only. A free-form picker can land in PR 13 if the founder asks.
- **Editable filter persistence.** Filters live in the URL; nothing is written to `localStorage` (density is the only persistent UI preference).
- **Cross-org audit views.** Slug-scoped; the OrgSwitcher unmounts and re-mounts the page.

### 3.2 Route nesting

`routes.tsx` currently has a single `/orgs/:slug/audit` leaf. Replace with a nested block:

```
/orgs/:slug/audit            -> AuditPage (renders SubTabBar + <Outlet />)
   index                     -> ActivityTab
   escalations               -> EscalationsTab
   traces                    -> TracesTab
   traces/:task_id           -> TracesTab (with selected task)
```

Sub-tabs are paths (not query string), so `g a` always lands on Activity and the browser back-button works between tabs.

### 3.3 Data flow

```
Daemon /orgs/{slug}/audit
    └─ lib/api/audit.listAudit({ task_id?, agent?, action?, since?, limit? })
        └─ providers/_real-audit.useAuditList(params)   [refetchInterval: 60_000]
            └─ hooks/audit.useAuditList                  [provider-routed]
                ├─ ActivityTab  (filters: agent, action, since, limit=200)
                ├─ EscalationsTab (filters: action in {escalation, escalation_resolved})
                └─ TracesTab    (filters: task_id; also reads useTaskRecall)
```

The hook signature mirrors `listAudit` with `slug` stripped — the active slug comes from `useParams` inside the provider, the same way `useTasksList` is wired.

### 3.4 Filter sidebar contract

`FilterSidebar` (existing pattern from PR 7) is rendered by `AuditPage` and shared across all three sub-tabs. Filter state is the source of truth:

```ts
interface AuditFilters {
  agent: string | null;
  action: string | null;
  since: '24h' | '7d' | 'all' | null;
  task_id: string | null;   // not in sidebar; set by Tasks deep link
}
```

Filters serialize to URL search params (`?agent=X&action=Y&since=24h&task_id=TASK-1`) so refresh + back/forward + deep links all work. `audit-filters.ts` owns the codec (parse → object, encode ← object). The Traces tab additionally uses `:task_id` as a URL segment when the founder picks a task from the canvas; the search-param form is the deep-link form and is preserved.

### 3.5 Filter-group composition per tab

Different tabs surface different filter groups so the sidebar is always context-aware (the underlying state is the same):

| Group   | Activity                          | Escalations                  | Traces                                |
| ------- | --------------------------------- | ---------------------------- | ------------------------------------- |
| AGENT   | shown                             | shown                        | shown (narrows the task picker)       |
| TYPE    | shown (all `audit_log` actions)   | hidden (pinned to escalation/resolved) | hidden                          |
| DATE    | shown (today / 7d / all)          | shown                        | shown                                 |
| TASK    | hidden — set via deep link only   | hidden                       | inline task picker in canvas, not the rail |

### 3.6 Expandable row pattern

Each Activity / Escalations row collapses to a single dense line and expands inline on click. Expansion shows:

- Full timestamp (RFC 3339 + relative).
- `payload` rendered as a JSON `<pre>` with copy-to-clipboard.
- Deep-link buttons: "View task" (if `task_id`), "View agent activity" (if `agent`).

Only one row is expanded at a time per tab; state is local to the tab component, not URL-encoded.

### 3.7 Trace tree

`TraceTree` is a fresh pattern, not a wrapper over `features/tasks/TaskRecallTree.tsx`. The two diverge in three ways:

1. **Cost annotations** — each node has a right-edge column showing the sum of `payload.token_usage.total` (and `payload.token_usage.total_cost_usd` if present) across all `session_end` entries for that task. The audit list is fetched once per page-view and grouped by `task_id` at render time.
2. **No mutation** — the recall tree on the Tasks page is interactive (clicking a node navigates the open detail pane); the trace tree is purely informational, with `IdBadge` linking out to the Tasks page.
3. **Density** — the trace renderer respects `useDensity()` for row gap; the recall tree does not.

`TaskRecallTree` keeps its current home; consolidation can wait until a third use appears (rule of three).

## 4. Components

### 4.1 `SubTabBar` (primitive)

Thin wrapper over `@radix-ui/react-tabs` that integrates with `react-router-dom`:

```tsx
<SubTabBar
  tabs={[
    { value: 'activity',    label: 'Activity',     to: routes.activity() },
    { value: 'escalations', label: 'Escalations',  to: routes.escalations() },
    { value: 'traces',      label: 'Traces',       to: routes.traces() },
  ]}
  active={activeTabId}
/>
```

Pinned at the top of the canvas; active tab inferred from the current pathname. Uses the same focus-ring and hover affordances as `primitives/Tabs.tsx`.

### 4.2 `AuditRow` (pattern)

Dense (32px) by default; comfortable mode adds 8px of vertical breathing room. Layout:

```
[id_badge?] [timestamp · mono_sm] [AgentChip] [action · mono_sm] [task_id_badge?] [>]
```

Click anywhere → toggles expansion. Keyboard: `Enter` toggles, `Esc` collapses. Expansion is rendered below the row inside the same `<li>` so density math stays correct.

### 4.3 `TraceTree` (pattern)

```tsx
<TraceTree
  root={recallNode}
  costs={costMap}            // task_id -> { tokens: number, usd?: number }
  density="comfortable" | "compact"
/>
```

Recursive children render at the same indent step (4 rem per level, clamped at 10) — same depth ladder used by `TaskRecallTree` so visual style stays consistent.

### 4.4 `AuditPage` (shell)

Three slots:

1. `FilterSidebar` (240px, left).
2. `SubTabBar` (top of canvas).
3. `<Outlet />` (1fr, scrolls).

Page-header line above the SubTabBar shows the active filter chips ("agent: content_writer ✕") for quick clearing. Filter changes call `setSearchParams` and let React Router rerender the appropriate tab.

### 4.5 `ActivityTab`

Renders `useAuditList({ agent, action, since, task_id, limit: 200 })` as a reverse-chronological list of `AuditRow`s. Empty state: "No audit entries match the current filters." A small "Load older" button at the bottom requests the next page (offset emulated client-side via `limit *= 2` until daemon offers pagination — sufficient for v1; documented as a follow-up).

### 4.6 `EscalationsTab`

Same data hook but **double-fetches**: one query for `action=escalation`, one for `action=escalation_resolved`. Result is folded into a table whose rows are escalation events with a "resolved" status pill if a matching `escalation_resolved` exists in the second query.

| Time | Agent | Task | Status | Resolved by | Δ |
| ---- | ----- | ---- | ------ | ----------- | - |

Sort defaults to most-recent escalation first. Clicking a row deep-links to Tasks (`useTasksRoutes().detail(task_id)`). Filters narrow by agent + since.

### 4.7 `TracesTab`

Left rail (inside the canvas): a slim task picker — list of unique `task_id`s from the audit query, displayed as `IdBadge + agent`. Right pane: `TraceTree` for the selected task, fed by `useTaskRecall(taskId)` and a per-task cost projection over the audit list.

URL-driven: clicking a task in the picker calls `navigate(routes.trace(taskId))` → renders `TracesTab` with `:task_id` in the URL. Cross-surface deep links land here.

## 5. Deep links (cross-surface)

The umbrella spec calls out deep links from Tasks, Agents, and Threads. URL shapes:

| Source                  | Link target                                                         | UI affordance               |
| ----------------------- | ------------------------------------------------------------------- | --------------------------- |
| `TaskDetailPane`        | `/orgs/:slug/audit?task_id=<task_id>`                               | "View audit →" link in the action row |
| `AgentsPage` row        | `/orgs/:slug/audit?agent=<agent_name>`                              | **deferred** — PR 8 not yet merged    |
| `ThreadsPage` detail    | `/orgs/:slug/audit?agent=<participants[0]>`                         | "Audit ↗" link in the header actions  |

Threads ↔ audit is intentionally narrow: thread events are not in `audit_log` v1, so we filter by the **lead participant** as a useful approximation rather than fabricating a thread-level filter the daemon doesn't support. The Agents row link is gated on PR 8 landing — the placeholder page has no rows to attach a link to. A follow-up commit adds it once the scorecard table is in place.

`AuditPage` reads `useSearchParams()` on mount; the filters initialize from there. `useEffect` syncs filter-state changes back to search params via `setSearchParams({ replace: true })` so the back button skips intermediate filter states.

## 9. Open questions (deferred)

- **Server-side pagination for the audit list.** v1 caps at `limit=200` client-side; the daemon route does not currently page. If founder usage hits the cap regularly, add `offset` to `query_audit_logs` and wire a TanStack `useInfiniteQuery`.
- **Free-form date range.** v1 ships today / this week / all. A `DateRangePicker` primitive (Radix Popover + two date inputs) is a PR 13 polish item.
- **Thread → audit deep link.** v1 filters by thread author. A richer projection (thread participants, time-windowed) requires the daemon to emit thread audit entries — out of scope.
- **Compose KB entry from an audit row.** Founder rulings flow through `happyranch kb add` in v1; an inline "promote to KB" affordance is a follow-up.
